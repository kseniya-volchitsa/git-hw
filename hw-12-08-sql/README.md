## Задание 1. Резервное копирование

Кейс

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

Приведите ответ в свободной форме.

--

### 1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день

**Рекомендуемый подход:** Комбинация полного (Full) и дифференциального (Differential) резервного копирования.

#### Схема работы:

    - Full Backup (еженедельно, например, в воскресенье) — полная копия всей базы данных.
    - Differential Backup (ежедневно) — копия всех изменений с момента последнего полного бэкапа.

#### Процесс восстановления за предыдущий день:

    - Восстановить последний полный бэкап.
    - Наложить последний дифференциальный бэкап (за предыдущий день).

#### Преимущества:

    - Быстрое восстановление (всего 2 шага).
    - Меньший размер, чем полный бэкап каждый день.
    - Простота реализации.

#### Инструменты:

    - PostgreSQL: pg_basebackup + WAL-архивация.
    - MySQL: mysqldump + mysqlbinlog или Percona XtraBackup.

--

### 1.2. Необходимо восстанавливать данные за час до предполагаемой поломки

**Рекомендуемый подход**: Полный бэкап + непрерывная WAL-архивация (Point-in-Time Recovery, PITR).

#### Схема работы:

    - Full Backup — выполняется периодически (например, раз в сутки).
    - WAL-архивация (Write-Ahead Log) — непрерывное сохранение всех изменений (журналов транзакций) с момента последнего полного бэкапа.

#### Процесс восстановления на час назад:

    - Восстановить последний полный бэкап.
    - "Накатить" все WAL-файлы до нужного момента времени (за час до поломки).

#### Преимущества:

    - Минимальная потеря данных (до нескольких секунд).
    - Гибкость — можно восстановиться на любой момент времени.
    - Автоматизация с помощью cron или специализированных инструментов.

#### Инструменты:

    - PostgreSQL: pg_basebackup + непрерывная архивация WAL + recovery.conf.
    - MySQL: Percona XtraBackup + бинарные логи (binlog) + mysqlbinlog для PITR.

--

### 1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных?

Да, возможен. Этот подход называется High Availability (HA) с автоматическим failover.

**Варианты реализации**:

| Метод | Описание | Время переключения |
|-------|----------|-------------------|
| **Master-Slave репликация с автоматическим промоутом** | При падении master один из slave автоматически становится master (с помощью VIP-адреса или ProxySQL). | 30-60 секунд |
| **Кластер с использованием DRBD/Pacemaker** | DRBD синхронизирует данные на уровне блоков. Pacemaker управляет ресурсами и автоматически переключает IP и сервисы. | 5-20 секунд |
| **SAN-кластер с общим хранилищем** | База данных работает на нескольких узлах, которые подключаются к общему хранилищу. При падении узла сервис перезапускается на другом узле. | 5-60 секунд |
| **Galera Cluster (MySQL/MariaDB)** | Синхронная многомастерная репликация. При падении одного узла другие продолжают работать. | ~1 секунда |
| **Patroni (PostgreSQL)** | Управление кластером PostgreSQL с автоматическим failover через etcd/Consul. | 10-30 секунд |

#### Требования:

    - Минимум два сервера (master + slave).
    - Настроенный мониторинг (Keepalived, Pacemaker, Patroni).
    - Разделяемый IP-адрес (VIP) или балансировщик (ProxySQL, HAProxy).

## Задание 2. PostgreSQL

2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

Приведите ответ в свободной форме.

--

### 2.1 Пример команды резервирования и восстановления (pg_dump / pg_restore)

**Резервное копирование (pg_dump)**

```bash
# Полный дамп базы данных в пользовательском формате
pg_dump -U postgres -d mydb -Fc -f /backup/mydb_$(date +%Y%m%d).dump

# Параметры:
# -U postgres  - пользователь
# -d mydb      - база данных
# -Fc          - пользовательский формат (сжатый, для pg_restore)
# -f           - файл для сохранения

# Полный дамп в SQL-формате
pg_dump -U postgres -d mydb -Fp -f /backup/mydb_$(date +%Y%m%d).sql

# Резервное копирование всех баз данных
pg_dumpall -U postgres -f /backup/all_databases_$(date +%Y%m%d).sql

``` 

**Восстановление данных (pg_restore)**

```bash
# Восстановление из пользовательского формата
pg_restore -U postgres -d mydb -v /backup/mydb_20250723.dump

# Параметры:
# -U postgres  - пользователь
# -d mydb      - целевая база
# -v           - подробный вывод

# Восстановление из SQL-файла
psql -U postgres -d mydb -f /backup/mydb_20250723.sql

``` 

**Восстановление с PITR (на момент времени)**

```bash
# 1. Восстанавливаем полный бэкап
sudo systemctl stop postgresql
rm -rf /var/lib/postgresql/data/*
pg_basebackup -U replicator -D /var/lib/postgresql/data -P

# 2. Создаём файл recovery.conf
cat > /var/lib/postgresql/data/recovery.conf << EOF
restore_command = 'cp /wal_archive/%f %p'
recovery_target_time = '2026-07-23 14:00:00'
EOF

# 3. Запускаем PostgreSQL
sudo systemctl start postgresql
```
--

### 2.1* Автоматизация резервного копирования PostgreSQL

**Да**, процесс можно автоматизировать.

#### Способы автоматизации:

1. **Cron (планировщик задач):**

```bash
# Добавляем в crontab
# Ежедневный полный бэкап в 02:00
0 2 * * * /usr/bin/pg_dump -U postgres -d mydb -Fc -f /backup/mydb_$(date +\%Y\%m\%d).dump

# Ежечасный WAL-бэкап
0 * * * * /usr/bin/rsync -av /var/lib/postgresql/wal_archive/ /backup/wal/

```

2. **Скрипт автоматизации:**

```bash
#!/bin/bash
# backup_postgres.sh
BACKUP_DIR="/backup/postgres"
DATE=$(date +%Y%m%d_%H%M%S)
DB_NAME="mydb"
pg_dump -U postgres -d $DB_NAME -Fc -f $BACKUP_DIR/${DB_NAME}_$DATE.dump
find $BACKUP_DIR -name "*.dump" -mtime +7 -delete  # Удаление бэкапов старше 7 дней

```

3. ** Использование готовых инструментов:**

        - pgBackRest — профессиональный инструмент для бэкапа PostgreSQL.
        - Barman — резервное копирование и восстановление PostgreSQL.
        - WAL-G — для архивации WAL и бэкапов в S3-совместимые хранилища.

## Задание 3. MySQL

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

Приведите ответ в свободной форме.

--

### 3.1 Пример команды инкрементного резервного копирования MySQL

MySQL не имеет встроенного инкрементного бэкапа. Для этой цели используется бинарный лог (binlog) и Percona XtraBackup (или MariaDB Backup).

#### Схема инкрементного бэкапа:

    - Полный бэкап — выполняется периодически (например, каждую неделю).
    - Инкрементные бэкапы — копия изменений с момента последнего бэкапа (любого типа).

**Команды (на примере Percona XtraBackup):**

```bash
# 1. Полный бэкап
xtrabackup --backup --target-dir=/backup/mysql/full

# 2. Инкрементный бэкап (только изменения)
xtrabackup --backup --target-dir=/backup/mysql/inc1 \
           --incremental-basedir=/backup/mysql/full

# 3. Следующий инкрементный бэкап
xtrabackup --backup --target-dir=/backup/mysql/inc2 \
           --incremental-basedir=/backup/mysql/inc1

# 4. Подготовка для восстановления (применение логов)
xtrabackup --prepare --apply-log-only --target-dir=/backup/mysql/full
xtrabackup --prepare --apply-log-only --target-dir=/backup/mysql/full \
           --incremental-dir=/backup/mysql/inc1
xtrabackup --prepare --target-dir=/backup/mysql/full \
           --incremental-dir=/backup/mysql/inc2
```

**Для MySQL без XtraBackup (бинарный лог):**

```bash
# Полный бэкап (mysqldump)
mysqldump -u root -p --all-databases --single-transaction --flush-logs > /backup/mysql/full_$(date +%Y%m%d).sql

# Инкремент — сохранение бинарных логов
mysqladmin -u root -p flush-logs
cp /var/lib/mysql/binlog.* /backup/mysql/binlog/

```
--

### 3.1* Преимущества реплики перед обычным резервным копированием

**Репликация** — это процесс непрерывной синхронизации данных с master на slave, что даёт ключевые преимущества:

| Сценарий | Репликация | Обычный бэкап |
|----------|------------|---------------|
| **Время восстановления (RTO)** | Минуты (переключение на slave) | Часы (восстановление из бэкапа) |
| **Потеря данных (RPO)** | Минимальная (секунды) | Часы (бэкап ежедневный) |
| **Нагрузка на master** | Частичная (slave берёт чтение) | Полная (бэкап нагружает master) |
| **Доступность** | Высокая (slave готов к работе) | Ниже (нужно восстанавливать) |
| **Тестирование изменений** | Можно на slave | Только на тестовой среде |
| **Резервирование** | Автоматическое | Требует ручного запуска |

### Вывод

Репликация — это **первый уровень защиты**, а бэкапы — **второй** (защита от случайного удаления, коррупции данных, человеческих ошибок). 

**Лучшая практика:** использовать репликацию для высокой доступности и периодические бэкапы для долгосрочного хранения.
