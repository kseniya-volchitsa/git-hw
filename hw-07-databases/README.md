### Задание 1. СУБД

#### 1.1. Бюджетирование проектов с финансовыми отчётами и прогнозированием рисков

**Рекомендуемая СУБД:** Реляционная (SQL) — **PostgreSQL** или **Oracle**.

**Обоснование:**  
Задачи требуют **целостности данных**, строгой структуры и сложных аналитических запросов. Реляционные СУБД обеспечивают:
- **ACID** — гарантию целостности при финансовых операциях.
- **Поддержку сложных JOIN** и агрегаций для аналитики.
- **Хранение финансовых данных** с высокой точностью (DECIMAL/NUMERIC).

**1.1* Ускорение хеширования:**  
Для ускорения хеширования можно использовать **аппаратное ускорение** (например, Intel QAT) или **специализированные библиотеки** (OpenSSL с оптимизациями под конкретную архитектуру). Также возможно применение **Redis** для кэширования хешей часто запрашиваемых данных.

---

#### 1.2. Лендинги и CRM для работы с лидами

**Для лендингов:** **NoSQL (документоориентированная)** — **MongoDB**.  
**Для CRM:** **Реляционная** — **PostgreSQL** или **MySQL**.

**Обоснование:**  
- **Лендинги** требуют гибкой схемы для хранения разнородных данных о посетителях (сессии, параметры, поведение). Документоориентированные СУБД позволяют быстро менять структуру.
- **CRM** требует структурированных данных о клиентах, сделках, истории взаимодействий. Реляционная СУБД обеспечивает целостность и сложные запросы (объединение данных).

**1.2* Одна СУБД:**  
Можно использовать **PostgreSQL** с типом данных `JSONB` для хранения гибких данных лендингов и реляционной структурой для CRM. Это даёт гибкость и структуру одновременно.

---

#### 1.3. База знаний по корпоративным нормам и правилам

**Рекомендуемая СУБД:** **Документоориентированная (NoSQL)** — **MongoDB** или **Elasticsearch**.

**Обоснование:**  
Документы, обучающие материалы, нормативы имеют иерархическую или нестрогую структуру. Документоориентированные СУБД позволяют хранить такие данные естественно, без необходимости жёсткой схемы.

**1.3* Использование существующей СУБД:**  
Можно использовать **PostgreSQL** с `JSONB` (из задачи 1.2), создав отдельную коллекцию/таблицу для документов. Это упростит инфраструктуру.

---

#### 1.4. Логистика: маршруты доставки и распределение курьеров

**Рекомендуемая СУБД:** **Графовая** — **Neo4j** или **ArangoDB**.

**Обоснование:**  
Задачи логистики — это классические задачи на **графы** (поиск кратчайшего пути, оптимизация маршрутов, связь между объектами и курьерами). Графовые СУБД идеально подходят для работы со связями.

**1.4* Отдел закупок:**  
Отдел закупок можно подключить к графовой СУБД через общие узлы (поставщики, материалы). Но для закупок (учёт заказов, склад) лучше использовать реляционную СУБД, связанную с графовой через ETL-процессы.

---

#### 1.5* Одна СУБД для всех задач

**Теоретически можно использовать** **PostgreSQL**:

- **Финансы** — реляционные таблицы с ACID.
- **Лендинги и CRM** — `JSONB` для гибких данных.
- **База знаний** — `JSONB` или полнотекстовый поиск.
- **Логистика** — через `WITH RECURSIVE` (рекурсивные запросы) можно реализовать графовые алгоритмы (но производительность будет ниже, чем у Neo4j).

**Ограничения:** PostgreSQL не оптимизирован под графовые запросы на больших данных. Для логистики лучше использовать специализированную графовую СУБД.

### Задание 2. Транзакции

#### 2.1. Пополнение баланса счёта телефона

**Пошаговый процесс (6 действий):**

1. **Начало транзакции** — `BEGIN TRANSACTION`.
2. **Проверка доступности средств** (у плательщика или системы).
3. **Списание средств** с источника (карта/счёт).
4. **Зачисление средств** на баланс телефона.
5. **Запись операции в историю** (лог).
6. **Фиксация транзакции** (`COMMIT`) — если все шаги успешны, или откат (`ROLLBACK`) при ошибке.

**2.1* Автоплатёж:**  
При автоплатеже шаги аналогичны, но инициатором выступает система, а не пользователь. Дополнительно добавляются проверки:  
- Активен ли автоплатёж.  
- Достаточно ли средств на привязанной карте.  
- Уведомление пользователя о списании.  
Остальные шаги (списание, зачисление, логирование) идентичны.

### Задание 3. SQL vs NoSQL

#### 3.1. Пять преимуществ SQL-систем перед NoSQL

| № | Преимущество | Пояснение |
|---|--------------|-----------|
| 1 | **Структурированные данные и строгая схема** | Гарантирует целостность и согласованность данных. |
| 2 | **ACID-транзакции** | Обеспечивают надёжность финансовых и критических операций. |
| 3 | **Сложные запросы** | Поддержка JOIN, подзапросов, агрегаций для аналитики. |
| 4 | **Зрелость и стандартизация** | Единый язык SQL, множество инструментов, большая экосистема. |
| 5 | **Поддержка внешних ключей** | Обеспечивает ссылочную целостность данных между таблицами. |

**3.1* Преимущества NewSQL:**  
NewSQL сочетает **ACID SQL-систем** с **горизонтальной масштабируемостью NoSQL**.  
- Высокая производительность при больших нагрузках.  
- Распределённые транзакции.  
- Поддержка стандартного SQL.  
- Автоматическое шардирование.  
- Устойчивость к отказам (как в NoSQL).  
Примеры: **Google Spanner, CockroachDB, TiDB**.

### Задание 4. Кластеры

**Критерии выбора СУБД для 1000 машин и больших вычислений:**

| Критерий | Описание |
|----------|----------|
| **Горизонтальная масштабируемость** | Возможность добавлять узлы без остановки. |
| **Поддержка распределённых транзакций** | Обеспечение согласованности данных. |
| **Модель данных** | Структурированные или неструктурированные данные. |
| **Требования к задержкам** | Низкая латентность или высокая пропускная способность. |
| **Устойчивость к отказам** | Автоматическое восстановление при сбоях. |

**Рекомендуемая модель распределённых вычислений:**  
**MapReduce** или **Apache Spark** (с поддержкой распределённых кластеров). Для реляционных данных — **Citus** (распределённый PostgreSQL). Для NoSQL — **Cassandra** или **ScyllaDB**.

**Обоснование:**  
При 1000 машинах важна **горизонтальная масштабируемость**. Системы типа **Cassandra** или **Citus** позволяют распределять данные и вычисления по кластеру, сохраняя отказоустойчивость. Для сложной аналитики (OLAP) лучше использовать **Greenplum** или **ClickHouse**.
